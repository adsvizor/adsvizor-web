# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

AdsVizor is a multi-tenant landing page + lead capture system. Each client gets a subdomain (e.g. `formations.adsvizor.com`) served by **Cloudflare Pages**, which auto-deploys on every `git push` to `main`. There is no build step — everything is vanilla HTML/CSS/JS.

A separate **Cloudflare Worker** (`cloudflare-worker.js`, deployed as `adsvizor-leads`) acts as a CORS proxy, forwarding lead POST requests from the browser to a Google Apps Script endpoint, which writes to Google Sheets.

## Deployment

- `git push` to `main` → Cloudflare Pages deploys static assets automatically
- Worker deployed via `npx wrangler deploy` — targets worker named `adsvizor-leads` (configured in `wrangler.jsonc`)
- Formation static pages: `node scripts/generate-formation-pages.js` → regenerates `clients/{slug}/pages/{category}/formation-*.html` + `sitemap.xml`
- Apps Script: copy-paste `apps-script/Code.gs` → Google Apps Script editor → Deploy new version

## Local development

Use `wrangler pages dev` (NOT Live Server) — required to run `functions/_middleware.js` locally:

```bash
npm run dev          # starts wrangler pages dev on port 5500
# or directly:
npx wrangler pages dev . --port 5500 --compatibility-date 2026-04-19
```

Then open `http://localhost:5500/?client=formations` for the landing page.
Client-specific pages work automatically: `http://localhost:5500/formations.html?client=formations`

> **Why not Live Server?** Client-specific pages are stored under `clients/{slug}/pages/` and `clients/{slug}/blog/`. The middleware routes them at runtime. Live Server has no middleware support.

## How the template system works

`index.html`, `blog.html`, `contact.html` are shared templates using `{{placeholder}}` syntax. At runtime, `script.js`:

1. Detects the client slug from the subdomain, or falls back to `?client=` URL param
2. Fetches `clients/{client_slug}/config.json`
3. Walks the DOM and replaces all `{{placeholders}}` in text nodes and attributes
4. Sets `document.body.classList.add("ready")` to reveal the page (FOUC prevention)

**Static formation pages** (`clients/{slug}/pages/formation-{slug}.html`) are generated by `scripts/generate-formation-pages.js` — they have hardcoded SEO content (title, meta, H1) for Google Ads Quality Score. Run the generator after changing formation content in `config.json`.

## File responsibilities

- `index.html` / `blog.html` / `contact.html` / `thank-you.html` / `privacy.html` — shared templates
- `landing.html` — served on `adsvizor.com` root domain (logo only, no client content)
- `main.css` — all styles; mobile-first, breakpoint at 768px
- `script.js` — config loading, placeholder rendering, UTM capture, multi-step form, analytics
- `cloudflare-worker.js` — CORS proxy for `/api/leads`; injects consent_ip, consent_user_agent, visitor_city, visitor_region from Cloudflare headers
- `wrangler.jsonc` — wrangler config, targets worker `adsvizor-leads`
- `_redirects` — Cloudflare Pages 301 redirects (formations-cpf → formations; old flat URLs → new category URLs)
- `sitemap.xml` — auto-generated by `generate-formation-pages.js`
- `apps-script/Code.gs` — Google Apps Script source (v6, 26 columns A–Z)
- `clients/{slug}/config.json` — all client-specific text and settings
- `clients/{slug}/agent.config.json` — blog agent prompts and nav links
- `clients/{slug}/pages/formations.html` — formations listing page
- `clients/{slug}/pages/{category}/formation-*.html` — static formation pages organised by category (bureautique, langues, management, marketing, finance, dev-personnel, entrepreneuriat, ia)
- `scripts/generate-formation-pages.js` — generates static pages + sitemap from config.json
- `functions/_middleware.js` — routes *.html to client directory; serves landing.html on root domain; handles 301 redirects via REDIRECTS_301 map (old flat URLs → new category URLs)

## Lead submission flow

Browser → POST JSON to Cloudflare Worker (`adsvizor-leads`) → Worker injects server-side fields → forwarded to Google Apps Script → appended to Google Sheet (26 columns A–Z).

**Worker injects (server-side, authoritative):**
- `consent_ip` — from `CF-Connecting-IP` header
- `consent_user_agent` — from `User-Agent` header
- `visitor_city`, `visitor_region` — from `request.cf` geolocation

**Frontend sends:**
- `consent_url`, `consent_text`, `consent_timestamp` — RGPD proof
- `formation_interest` — from dropdown or hidden input
- Standard UTM fields, professional_status, etc.

Multi-step form: Step 1 (name/phone/email/consent) → partial send on "Continuer" → Step 2 (status/formation) → full send on submit → redirect to `thank-you.html?code=XXXXXX`.

## Webuilder pipeline

Automated system to generate a new client site from an email.

**Trigger:** Send email to webuilder@adsvizor.com with subject `WEBUILDER: {slug} - Description`, body with client info, and optional PDF/DOCX/XLSX catalog attachment.

**Flow:** Email → Apps Script (`apps-script/IntakeAgent.gs`) → GitHub branch `webuilder/{slug}` + PR → GitHub Actions (`webuilder-agent.yml`) → Claude API generates `clients/{slug}/config.json` → merge PR → Cloudflare Pages deploys → `webuilder-dns.yml` configures DNS + custom domain automatically.

**Key files:**
- `scripts/webuilder-agent.js` — Node.js agent: extracts catalog text, searches Brave API for reviews + competitor sites, calls Claude claude-opus-4-6 (max_tokens: 12000) to generate complete config.json
- `.github/workflows/webuilder-agent.yml` — triggers on PR open for `webuilder/*` branches
- `.github/workflows/webuilder-dns.yml` — triggers on push to main when `clients/*/config.json` changes; also has `workflow_dispatch` for manual trigger with slug input

**GitHub secrets required:**
- `ANTHROPIC_API_KEY`, `CLOUDFLARE_API_TOKEN`, `CLOUDFLARE_ZONE_ID`, `CLOUDFLARE_ACCOUNT_ID`, `BRAVE_API_KEY`

**config.json schema** — all fields must be present (see `clients/formations/config.json` as reference):
- SEO/meta, nav (5 items), hero, stats (3), benefits (2), why_us (a–d), testimonials, service dropdown (field_formation_opt_*), form personal fields (first/last name, email, phone, status, message, submit, disclaimer), thank-you page, contact page (steps 1–5 with icon/image/title/text), blog page + 4 posts (with date/tag), privacy page, footer

## CSS / JS cache busting

Every change to `main.css` or `script.js` requires bumping the version query param in all HTML files, otherwise Cloudflare serves stale cached files.

Current versions: `main.css?v=15`, `script.js?v=27`

```bash
# Bump CSS (replace N with current version)
find . -name "*.html" | grep -v "/blog/" | xargs sed -i 's|main\.css?v=N|main.css?v=N+1|g'
echo "/* vN+1 */" >> main.css   # also change file content to bust Cloudflare edge cache

# Bump JS
find . -name "*.html" | grep -v "/blog/" | xargs sed -i 's|script\.js?v=N|script.js?v=N+1|g'
```

**IMPORTANT — always verify JS syntax after editing script.js:**
```bash
node --check script.js
```
A syntax error silently breaks every page (body stays `visibility:hidden` forever).

## File editing safety rules

- **Never use the Edit tool on `script.js` or `generate-formation-pages.js`** — these files contain multi-byte UTF-8 characters (emoji) that cause the Edit tool to corrupt or truncate the file.
- **Always use Python** (`open(..., encoding='utf-8')`) or `sed` for edits to these files.
- After every JS edit: run `node --check script.js` before committing.

## Git commit workflow

The pre-commit hook blocks commits if any HTML file doesn't end with `</html>`. Blog files in `clients/formations/blog/` are pre-existing truncated files that trigger this. Always exclude them:

```bash
git add -A
git restore --staged clients/formations/blog/
git commit -m "..."
git push
```

## FOUC prevention / page visibility

All pages use this pattern to prevent flash of unstyled content:

```html
<style>body{visibility:hidden}</style>
<link rel="preload" href="/main.css?v=15" as="style" onload="this.onload=null;this.rel='stylesheet'" />
<noscript><link rel="stylesheet" href="/main.css?v=15" /></noscript>
```

`script.js` calls `document.body.classList.add("ready")` once config is loaded and placeholders are replaced. The CSS rule `body.ready { visibility: visible }` reveals the page.

## Google Analytics — page_view

All gtag config calls use `send_page_view: false` to prevent GA from recording `{{placeholder}}` as the page title. A manual `page_view` event fires after placeholder replacement:

```js
gtag('config', 'GT-KD7C7TR3', { send_page_view: false });
// ... later in script.js after body.ready:
gtag('event', 'page_view', { page_title: document.title, page_location: window.location.href });
```

## Mobile CTA button (btn-header-cta)

On mobile, a red CTA button appears **below the H1 in the hero section** (not in the header). It is hidden on desktop (`display: none !important` at `>=768px`) since the lead form is always visible in the right column.

- Color: `#c2410c` (WCAG AA contrast 5:1 on white)
- Injected by `initHeaderCta()` in `script.js` after the H1 (`section.hero h1`)
- On pages where the button is hardcoded in HTML (`formations.html`, `blog.html`, `index.html`), `initHeaderCta()` skips injection but still wires the smooth-scroll click listener
- Uses `window.location.pathname + "#contact"` as href (NOT `"#contact"`) to avoid `<base href="/">` conflicts on generated pages
- Old `.formation-mobile-cta` buttons are hidden globally via `main.css`: `.formation-mobile-cta { display: none !important; }`

## Static contact page

`clients/formations/pages/contact.html` is a static pre-rendered version of `contact.html` generated by a Python script that replaces all `{{placeholders}}` from `config.json`. It allows Google crawler to read the contact page content without executing JS. Regenerate it after changing contact-related keys in `config.json`.

## Formation nav_label

In `config.json`, formations can have a `nav_label` field to control the short text shown in the pill navigation links (separate from the full SEO `title`). Example: `"nav_label": "Anglais TOEIC"`. The generator prefers `nav_label` over a truncated `title`.
