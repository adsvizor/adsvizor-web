# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

AdsVizor is a multi-tenant landing page + lead capture system. Each client gets a subdomain (e.g. `formations.adsvizor.com`) served by **Cloudflare Pages**, which auto-deploys on every `git push` to `main`. There is no build step ‚Äî everything is vanilla HTML/CSS/JS.

A separate **Cloudflare Worker** (`cloudflare-worker.js`, deployed as `adsvizor-leads`) acts as a CORS proxy, forwarding lead POST requests from the browser to a Google Apps Script endpoint, which writes to Google Sheets.

## Deployment

- `git push` to `main` ‚Üí Cloudflare Pages deploys static assets automatically
- Worker deployed via `npx wrangler deploy` ‚Äî targets worker named `adsvizor-leads` (configured in `wrangler.jsonc`)
- Formation static pages: `node scripts/generate-formation-pages.js` ‚Üí regenerates `clients/{slug}/pages/{category}/formation-*.html` + `sitemap.xml`
- Apps Script: copy-paste `apps-script/Code.gs` ‚Üí Google Apps Script editor ‚Üí Deploy new version

## Local development

Use `wrangler pages dev` (NOT Live Server) ‚Äî required to run `functions/_middleware.js` locally:

```bash
npm run dev          # starts wrangler pages dev on port 5500
# or directly:
npx wrangler pages dev . --port 5500 --compatibility-date 2026-04-19
```

Then open `http://localhost:5500/?client=formations` for the landing page.
Client-specific pages work automatically: `http://localhost:5500/formations.html?client=formations`

> **Why not Live Server?** Client-specific pages are stored under `clients/{slug}/pages/` and `clients/{slug}/blog/`. The middleware routes them at runtime. Live Server has no middleware support.

## How the template system works

`index.html`, `blog.html`, `contact.html` are shared templates using `{{placeholder}}` syntax. At runtime, `script.js`:

1. Detects the client slug from the subdomain, or falls back to `?client=` URL param
2. Fetches `clients/{client_slug}/config.json`
3. Walks the DOM and replaces all `{{placeholders}}` in text nodes and attributes
4. Sets `document.body.classList.add("ready")` to reveal the page (FOUC prevention)

**Static formation pages** (`clients/{slug}/pages/formation-{slug}.html`) are generated by `scripts/generate-formation-pages.js` ‚Äî they have hardcoded SEO content (title, meta, H1) for Google Ads Quality Score. Run the generator after changing formation content in `config.json`.

## File responsibilities

- `index.html` / `blog.html` / `contact.html` / `thank-you.html` / `privacy.html` ‚Äî shared templates
- `landing.html` ‚Äî served on `adsvizor.com` root domain (logo only, no client content)
- `main.css` ‚Äî all styles; mobile-first, breakpoint at 768px
- `script.js` ‚Äî config loading, placeholder rendering, UTM capture, multi-step form, analytics
- `cloudflare-worker.js` ‚Äî CORS proxy for `/api/leads`; injects consent_ip, consent_user_agent, visitor_city, visitor_region from Cloudflare headers
- `wrangler.jsonc` ‚Äî wrangler config, targets worker `adsvizor-leads`
- `_redirects` ‚Äî Cloudflare Pages 301 redirects (formations-cpf ‚Üí formations; old flat URLs ‚Üí new category URLs)
- `sitemap.xml` ‚Äî auto-generated by `generate-formation-pages.js`
- `apps-script/Code.gs` ‚Äî Google Apps Script source (v6, 26 columns A‚ÄìZ)
- `clients/{slug}/config.json` ‚Äî all client-specific text and settings
- `clients/{slug}/agent.config.json` ‚Äî blog agent prompts and nav links
- `clients/{slug}/pages/formations.html` ‚Äî formations listing page
- `clients/{slug}/pages/{category}/formation-*.html` ‚Äî static formation pages organised by category (bureautique, langues, management, marketing, finance, dev-personnel, entrepreneuriat, ia)
- `scripts/generate-formation-pages.js` ‚Äî generates static pages + sitemap from config.json
- `functions/_middleware.js` ‚Äî routes *.html to client directory; serves landing.html on root domain; handles 301 redirects via REDIRECTS_301 map (old flat URLs ‚Üí new category URLs)

## Form versions (feature flag)

script.js supports four form variants, selected by priority:

1. **`?form=N` URL param** ‚Äî overrides everything; useful for A/B testing without deploying
2. **`config.active_form`** in `clients/{slug}/config.json` ‚Äî production switch (e.g. `"active_form": "4"`)
3. **`data-form="3"` on `<form>`** ‚Üí form3 (default when no flag set)
4. **`data-simple="true"` on `<form>`** ‚Üí form2
5. **(no attribute)** ‚Üí form1 (classic 2-step)

| Version | Flow | Key difference |
|---------|------|----------------|
| form1 | Step1 (name/phone/email/consent) ‚Üí Step2 (status/formation) | Classic multi-step |
| form2 | Single-step | Simple |
| form3 | 0 (consent) ‚Üí 1 (formation) ‚Üí 2 (status) ‚Üí 3a/3b (result) ‚Üí 4 (coords) | Full funnel |
| form4 | 1 (formation) ‚Üí 2 (status) ‚Üí 3a/3b (result) ‚Üí 4 (coords + consent) | No intro; consent at end; no email |

To activate form4 site-wide: add `"active_form": "4"` to `clients/formations/config.json`.
To test locally without deploying: append `?form=4` to any URL.
To revert: remove or change `active_form` in config.json.

## Key config.json fields

- `client_slug`, `offer_id`, `page_version` ‚Äî included in lead payloads
- `form_action` ‚Äî the Apps Script endpoint URL
- `nav_item_0_href` / `nav_item_0_label` ‚Äî Accueil (home) link
- `nav_item_1_href` through `nav_item_4_href` ‚Äî nav items
- Blog keys: `post_1_*` through `post_10_*`
- Why-us section: `why_us_title`, `why_us_a_*` through `why_us_d_*`
- Formation dropdown: `field_formation_label`, `field_formation_opt_*`, `field_formation_permis_msg`
- Contact process: `contact_step_1_*` through `contact_step_5_*` (with `_image`, `_icon`, `_title`, `_text`)

## Lead submission flow

Browser ‚Üí POST JSON to Cloudflare Worker (`adsvizor-leads`) ‚Üí Worker injects server-side fields ‚Üí forwarded to Google Apps Script ‚Üí appended to Google Sheet (26 columns A‚ÄìZ).

**Worker injects (server-side, authoritative):**
- `consent_ip` ‚Äî from `CF-Connecting-IP` header
- `consent_user_agent` ‚Äî from `User-Agent` header
- `visitor_city`, `visitor_region` ‚Äî from `request.cf` geolocation

**Frontend sends:**
- `consent_url`, `consent_text`, `consent_timestamp` ‚Äî RGPD proof
- `formation_interest` ‚Äî from dropdown or hidden input
- Standard UTM fields, professional_status, etc.

Multi-step form: Step 1 (name/phone/email/consent) ‚Üí partial send on "Continuer" ‚Üí Step 2 (status/formation) ‚Üí full send on submit ‚Üí redirect to `thank-you.html?code=XXXXXX`.

## Webuilder pipeline

Automated system to generate a new client site from an email.

**Trigger:** Send email to webuilder@adsvizor.com with subject `WEBUILDER: {slug} - Description`, body with client info, and optional PDF/DOCX/XLSX catalog attachment.

**Flow:** Email ‚Üí Apps Script (`apps-script/IntakeAgent.gs`) ‚Üí GitHub branch `webuilder/{slug}` + PR ‚Üí GitHub Actions (`webuilder-agent.yml`) ‚Üí Claude API generates `clients/{slug}/config.json` ‚Üí merge PR ‚Üí Cloudflare Pages deploys ‚Üí `webuilder-dns.yml` configures DNS + custom domain automatically.

**Key files:**
- `scripts/webuilder-agent.js` ‚Äî Node.js agent: extracts catalog text, searches Brave API for reviews + competitor sites, calls Claude claude-opus-4-6 (max_tokens: 12000) to generate complete config.json
- `.github/workflows/webuilder-agent.yml` ‚Äî triggers on PR
